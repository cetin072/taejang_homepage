# Phase 1 통합 검수·시범운영 준비

상태: **Draft PR 검수 기록 — 운영 Supabase에는 적용하지 않는다.**

## 기준과 구현 범위

이 검수는 `PROJECT_CHARTER.md`, MVP·역할·RLS·일반 근로자 기획 문서와 Phase 1A, Today, 작업방법, 일정·공지, 역할 셸, 자주 보는 안내 구현 문서를 기준으로 한다. 새 업무 기능을 추가하지 않고 인증·계정, 일반 근로자 5개 읽기 화면, 역할별 셸·관리 화면을 하나의 시범운영 흐름으로 점검한다.

| 영역 | 구현 상태 | 검수 결과 |
| --- | --- | --- |
| 인증·승인 | 가입은 `pending`, 최고관리자 승인 후 `active` | 비운영 수동 확인 필요 |
| 계정 상태 | `pending`, `active`, `suspended`, `departed`, `deleted` | 매 DB 요청에서 현재 상태 재확인 |
| 일반 근로자 | Today, 작업방법, 내 일정, 중요공지, 자주 보는 안내 | 모바일 단일 열·600px 이하 PC 폭 |
| 관리자 | 일정·공지·작업방법·안내 관리 | 4개 관리자 역할만 동적 로드 |
| 계정 관리 | 최고관리자 전용 승인·상태·역할 | 마지막 활성 최고관리자 보호 |

실제 역할 코드는 `super_admin`, `ceo`, `operations_manager`, `department_lead`, `field_lead`, `worker_support_lead`, `worker_support_staff`, `promotion_lead`, `promotion_staff`, `office_staff`, `general_worker`, `work_assistant`, `external_guide`다. 역할은 URL이 아니라 서버 접근 문맥과 고정 우선순위로 정한다.

## 역할별 동선

| 역할 | 로그인 후 | 핵심 동선 | 제한 |
| --- | --- | --- | --- |
| 최고관리자 | 시스템 관리 대시보드 | 승인 대기 → 계정 승인 → 운영 관리 화면 | 마지막 최고관리자 보호 |
| 대표이사 | 대표이사 요약 | 중요공지·주요 일정 확인 | 계정·작성 관리 메뉴 없음 |
| 운영총괄 | 운영총괄 대시보드 | 일정·공지·작업방법·안내 관리 | 계정 상태·역할 변경 불가 |
| 부서 팀장 | 부서 운영 대시보드 | 본 부서 대상 자료 관리 | 타 부서·범위 밖 대상 차단 |
| 현장 책임자 | 현장 실행 홈 | 담당 작업반 자료 관리 | 전사·타 작업반 작성 차단 |
| 전문·사무직 | 업무 안내 | 본인 대상 자료 조회 | 관리자 메뉴 없음 |
| 일반 근로자 | 오늘 필요한 정보 | 5개 읽기 메뉴 → 로그아웃 | 사이드바·작성 기능 없음 |

`/app/` 직접 진입, 새로고침, 브라우저 뒤로가기는 `get_my_access_context()`를 다시 확인한다. 세션 없음·비활성 계정은 `/staff/`로 돌아간다.

## 권한·UX·접근성 결과

- 일반 근로자는 관리자 RPC와 직접 테이블 쓰기가 차단된다.
- 회사·현재 부서·현재 유효 작업반·본인 개인 대상만 읽기 RPC에서 반환한다.
- 타 부서·타 작업반·타인, 초안, 사용 중지, 게시·적용기간 밖 자료는 목록·상세·직접 ID 접근에서 차단한다.
- 기존 대상과 변경 대상 모두 서버에서 재검증하며 security-definer 함수는 빈 `search_path`를 사용한다.
- 감사로그에는 상태·범위·변경 여부만 기록하고 장문 본문·토큰·민감정보를 넣지 않는다.
- 일반 근로자 화면은 단일 열, 큰 버튼, 쉬운 오류·빈 상태, `aria-pressed` 필터를 사용한다. 관리자 셸은 `aria-current`, `aria-expanded`, `aria-controls`, 키보드 포커스와 reduced motion 기준을 사용한다.

자동 검사는 구조·ARIA·비밀값 부재·안전 링크를 확인한다. 실제 기기, 200% 확대, 키보드, 스크린리더는 수동 검수로 분리한다.

## 시간대·migration

적용 순서: Phase 1A → Today → 작업방법 → 일정·공지 → 자주 보는 안내 → `20260724000200_phase1_readiness_kst_dates.sql`.

마지막 migration은 `korea_current_date()`로 KST 기준을 고정해 일정 기본 조회일, 작업반 멤버십 기준일, 안내 적용기간, 공지 확인 대상 집계를 정리한다. 앱도 Today·일정 조회에 KST 날짜를 명시한다. migration은 깨끗한 비운영 DB에서 순서대로 한 번 적용하는 전제이며 운영 DB 재실행은 금지한다.

## 발견 문제와 조치

| 등급 | 문제 | 조치 |
| --- | --- | --- |
| 차단 | 없음 확인 | CI·비운영 수동 검수 성공 전 시범운영 시작 금지 |

| 낮음 | 작업방법 관리는 기존 Today 관리 패널을 사용 | 별도 화면 분리는 후속 UX 개선 |

## 시범운영 판단과 후속

**조건부 가능.** Draft PR의 CI와 Deploy Preview가 성공하고, 비운영 샘플 데이터·실제 기기 수동 검수·초기 최고관리자 확인을 마친 경우에만 제한된 시범운영을 시작한다. 운영 적용은 별도 사용자 승인 사항이다.

우선순위는 (1) 비운영 역할별·KST 경계 수동 확인, (2) 최소 2명 최고관리자와 세션 종료 절차 준비, (3) 대표이사 상세·보고 기능 별도 명세, (4) 시범운영 UX 피드백 반영 순서다.
