# Phase 1B 콘텐츠 중심 운영·UX 프로토타입

> 상태: **정적 프로토타입 — 실제 운영 시스템 아님**

## 1. 운영 목적

태장 홈페이지는 공식 기업 홈페이지이자 중요 공지와 공식 콘텐츠를 게시하는 공간이다. 동시에 네이버 블로그, 인스타그램, 유튜브, 언론기사 등 외부 채널의 콘텐츠를 모아 보여주는 통합 아카이브이며, 시간이 지나며 글과 사진이 축적되는 온라인 기록관이다.

홍보팀은 블로그·인스타그램을 자주 게시하고 유튜브는 주기적으로 게시한다. 홈페이지 자체 게시와 외부 콘텐츠 카드는 필요할 때 수동으로 등록한다.

## 2. 우선순위 변경

Phase 1A는 보안 위험·상태·승인·정적 발행 정책을 확인한 기술 프로토타입이다. Phase 1B는 실제 직원 업무에 맞게 범위를 조정한 콘텐츠 UX 프로토타입이다. 따라서 첫 화면에서 직원이 바로 이해할 수 있는 글 작성, 외부 카드 등록, 이미지 등록, 임시저장, 미리보기, 검색을 우선한다. 다단계 결재, 상세 퇴사 재배정 UI, 실제 세션 강제 종료와 RLS는 운영 연결 단계로 보류한다. Phase 1A는 폐기하지 않으며 실제 보안 구현의 참고자료로 유지한다.

## 3. 콘텐츠 구분

| 유형 | 예시 | 공개 방식 |
| --- | --- | --- |
| 홈페이지 자체 게시물 | 공지사항, 태장소식, 행사·포스터, 채용공고, 보도자료 | 홈페이지 상세 화면에서 전체 내용을 읽음 |
| 외부 콘텐츠 카드 | 네이버 블로그, 인스타그램, 유튜브, 언론기사, X | 제목·썸네일·요약·출처·날짜를 카드로 표시하고 원문으로 이동 |

## 4. 일상 운영 흐름

1. 홍보팀이 홈페이지 글 또는 외부 콘텐츠 등록을 선택한다.
2. 초안 입력 중 자동저장 상태를 확인한다.
3. 이미지·게시일·공개 상태를 확인하고 미리보기한다.
4. 게시 요청을 제출한다.
5. 단순 검토자가 확인한 뒤 게시 또는 수정 요청한다.
6. 게시된 글과 카드가 통합 목록과 공개 아카이브에 쌓인다.

## 5. 사용자 유형

- 작성자: 글·외부 카드를 만들고 임시저장·미리보기·게시 요청을 수행한다.
- 검토자: 게시 요청을 확인하고 승인 또는 수정 요청을 한다.
- 관리자: 기본 계정 활성·정지, 콘텐츠 운영 설정을 관리한다.
- 최고관리자: Phase 1A에서 정의한 권한·보안 구조를 향후 실제 연결 시 관리한다.

## 6. MVP 기능

- 관리자 홈에서 두 등록 방식을 명확히 선택
- 자체 글 작성과 행사·포스터 조건부 필드
- 외부 링크 카드 등록 및 제안값 수정
- 대표 이미지·본문 이미지·첨부파일 자리
- 임시저장·자동저장 상태·미리보기
- 통합 목록 검색·필터·복제·숨김·확인창이 있는 삭제 데모
- 데스크톱/모바일 미리보기
- 단순 게시 요청·승인 상태 표현

## 7. 보류 기능

다단계 결재, 실제 세션 강제 종료, 상세 퇴사자 재배정, 실제 RLS/Storage, SNS 자동 수집, 예약 발행, 통계·다국어·AI 기능은 실제 운영 요구가 확인된 뒤 단계적으로 결정한다.

## 8. Phase 1A 보안 원칙 유지

운영 연결 시에도 active 계정만 민감 작업을 수행하고, suspended/departed 상태는 요청별로 즉시 차단한다. 세션 폐기는 별도 방어로 유지하며, 승인된 revision만 정적 발행한다. 원본·감사·내부 메모는 공개 결과물에서 제외하고, 장애·발행 실패 시 마지막 정상 공개본을 유지한다. 이번 화면은 이를 구현한 것이 아니라 업무 흐름을 설명하는 정적 모형이다.

## 9. 관리자 정보구조

```text
관리자 홈
├─ 외부 콘텐츠 빠른 등록
├─ 홈페이지 글 작성
├─ 임시저장 / 게시 요청
├─ 최근 게시물
└─ 콘텐츠 목록
   ├─ 검색·필터
   ├─ 열기·수정·미리보기·복제
   └─ 숨김·휴지통 이동 데모
```

## 10. 공개 표시 구조

자체 글은 제목·게시일·본문·대표 이미지·첨부파일·이전/다음 글을 제공한다. 외부 카드는 출처 배지, 썸네일, 제목, 요약, 날짜, 카테고리, 원문 보기 버튼을 제공한다. 두 유형은 카드에서 섞여 보일 수 있지만 출처 배지로 명확히 구분한다.

## 11. 향후 Supabase 연결 범위

Supabase는 아직 최종 채택되지 않았다. 채택 시 먼저 콘텐츠·revision·검토·승인·파일 메타데이터와 계정 상태를 연결하고, 그 다음 요청별 active 검사·RLS·Storage 정책·정적 발행 작업을 단계적으로 검증한다. 공개 홈페이지가 DB를 실시간 조회하는 구조는 우선안이 아니다.

## 12. 사진·영상과 무료 플랜 정책

원본 사진·영상은 공개 저장소에 넣지 않고 제한된 원본 보관 영역에서 관리한다. 공개용 파생 이미지와 필요한 PDF만 승인 후 발행한다. 무료 플랜을 오래 사용하려면 원본 동영상 업로드를 피하고, 웹용 압축 이미지·명확한 파일명·중복 제거·필요한 보존기간을 적용한다. 실제 저장소와 보존기간은 사용자 승인 사항이다.

## 13. 프로토타입 범위

`admin/` 아래 정적 HTML/CSS/JavaScript만 사용한다. 외부 API, Supabase SDK, OAuth, 서버 업로드, SNS 자동 수집은 없다. 임시저장 데모는 `localStorage`에만 기록되며 화면 상단에 이 사실을 표시한다. 실제 계정·DB·파일은 존재하지 않는다.

## 14. 이후 단계

1. 직원 몇 명과 화면 용어·등록 흐름을 검토한다.
2. 필요한 필드와 게시 승인 수준을 확정한다.
3. 사용자 승인 후 실제 데이터 모델·Auth·Storage·정적 발행 방식을 결정한다.
4. Phase 1A 보안 원칙을 실제 서비스 경계에 단계적으로 적용한다.
5. 운영 데이터와 공개 배포를 분리한 검증 환경에서 테스트한다.

## 15. 사용자 승인 필요 사항

Supabase 최종 채택, 관리자 앱 배포 위치, 파일 저장소와 무료 플랜 보존정책, 승인자 분리 수준, 예약 발행, 외부 카드의 필수 메타데이터, 공개 홈페이지 반영 방식은 이 프로토타입만으로 확정하지 않는다.
