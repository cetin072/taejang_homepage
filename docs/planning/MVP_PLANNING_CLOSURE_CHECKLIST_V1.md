# 태장 업무플랫폼 MVP 기획 종료 점검표 v1

상태: **검토 중**

## 1. 목적

이 문서는 태장 업무플랫폼의 초기 MVP 기획이 구현 단계로 넘어갈 만큼 충분히 정리되었는지 확인한다. 새 기능을 추가하기 위한 문서가 아니라, 이미 확정된 범위의 누락·충돌·과도한 복잡성을 점검하기 위한 문서다.

## 2. 현재 확정된 핵심 방향

- 공개 홈페이지와 내부 업무플랫폼의 장애 경계를 분리한다.
- 일반 장애인 근로자 화면은 읽기·확인 중심 정보게시판이다.
- 일반 근로자에게 시작·완료·실적·출퇴근·자유보고 입력을 요구하지 않는다.
- 생산·물류·환경정비·농장 등 현장업무는 반장·관리자가 작업반 단위로 운영하고 예외만 기록한다.
- 현장부서는 최소기록, 일반 사무업무는 기본 업무기록, 지원·상담·홍보·기획은 전문 상세기록을 적용한다.
- 기록이 어려운 직원에게는 단계별 질문·선택항목·예시·누락 확인·팀장 보완 기능을 제공한다.
- 근로자지원은 감시·평가가 아니라 적응·지원·고용유지를 목적으로 한다.
- 승인 기본선은 직원 → 팀장 → 운영총괄이며, 운영총괄이 대표이사 상신 여부를 판단한다.
- Google Workspace는 초기에는 링크 중심으로 사용하고 복잡한 API·OAuth 연동은 미룬다.
- 기능 추가보다 누락 방지, 상태 가시성, 현장 사용성과 개인정보 보호를 우선한다.

## 3. 역할·권한 점검

- [x] 대표이사, 운영총괄, 팀장·반장, 일반 직원, 근로자지원 담당자, 외부지도자 역할 원칙이 문서화되었는가
- [x] 일반 근로자에게 관리·결재·민감정보 기능이 노출되지 않는가
- [x] 외부지도자의 접근 범위가 본인 일정·업무·준비사항으로 제한되는가
- [x] 근로자지원 기록과 현장 전달사항의 접근권한이 분리되는가
- [x] 마지막 최고관리자 보호와 퇴사자 즉시 차단 원칙이 있는가
- [ ] 실제 조직의 총괄반장·업무반장·현장 관리자 담당자를 운영 단계에서 지정했는가

실제 담당자 이름 지정은 시스템 개발 착수의 필수조건이 아니며, 계정 생성 전 운영결정으로 처리할 수 있다.

## 4. 일반 근로자 화면 점검

- [x] 오늘의 업무, 작업방법, 내 일정, 중요공지, 자주 보는 안내가 정의되었는가
- [x] 작업 중 상시 휴대전화 사용을 유도하지 않는가
- [x] 개인기기와 공용 태블릿의 표시범위가 구분되는가
- [x] 사진·쉬운 문장·큰 글자·고대비 등 접근성 원칙이 있는가
- [x] 장애명 대신 실제 지원방식으로 화면을 조정하는가
- [x] 시작·완료·수량·실적·출퇴근 입력이 제외되었는가
- [x] 야외활동 등 특별일정 안내에 필요한 항목이 정의되었는가

## 5. 현장 운영 점검

- [x] 반복 업무 템플릿 항목이 정의되었는가
- [x] 템플릿과 당일 업무 정보가 분리되는가
- [x] 작업반 단위 배정과 예외 근로자 별도처리 원칙이 있는가
- [x] 작업 결과, 예외·변경, 문제·조치, 인계의 최소기록 유형이 정의되었는가
- [x] 작업방법 자료의 사진·3~7단계·완성 예시·주의사항·변경이력이 정의되었는가
- [x] 현장 안전과 문제 대응이 플랫폼보다 우선하는가
- [ ] 실제 생산·물류 반복 업무 템플릿 1~2개를 현장 확인 후 등록했는가

실제 템플릿 샘플은 개발 전 데이터 모델 검증용으로 준비하면 되며 기획 종료를 막지 않는다.

## 6. 전문업무 기록 점검

- [x] 현장 최소기록이 모든 부서에 일률적으로 적용되지 않는가
- [x] 근로자지원·상담 기록의 상세항목과 제한 접근 원칙이 있는가
- [x] 홍보·기획·외부사업 기록의 목적·근거·결과·후속조치 기준이 있는가
- [x] 기록이 어려운 직원에게 구조화된 입력을 제공하는가
- [x] 팀장·책임자가 내용 누락을 보완하는 검토자 역할을 맡는가
- [x] 민감정보와 현장 전달정보가 분리되는가

## 7. 승인·협업 점검

- [x] 기본 승인 흐름과 운영총괄 최종승인 원칙이 정의되었는가
- [x] 대표이사 상신 기준이 중요 계약·인사·예산·신사업·중대위험 등으로 제한되는가
- [x] 부서 간 요청은 수신부서 팀장이 수락·조정·배정하는가
- [x] 요청 작성 예시와 완료기준 안내 원칙이 있는가
- [x] 업무 인계와 과거 기록 조회 원칙이 있는가

## 8. 기술·보안 점검

- [x] 초기 MVP는 직접입력, 링크, 상태표시, 누락방지를 우선하는가
- [x] Google Workspace의 초기 범위가 링크 중심으로 제한되는가
- [x] 건강·장애·인사 민감정보의 최소수집·분리 원칙이 있는가
- [x] 계정 상태 active / suspended / departed / deleted가 정의되었는가
- [x] 퇴사자 접근차단·세션무효화·작성차단·기록보존·재배정 원칙이 있는가
- [x] 일반 직원의 계정상태 변경과 최종승인 권한이 제한되는가
- [ ] 최종 데이터 모델·RLS·화면별 API 명세가 작성되었는가

최종 데이터 모델과 API 명세는 기획 종료 후 Phase 0 기술설계에서 작성한다.

## 9. 초기 MVP에서 제외된 복잡성

- AI 업무배정·평가·상담판단
- 직원별 생산성·숙련도·위험도 점수
- 실시간 위치·생체인증·GPS 출퇴근
- 일반 근로자 상시 알림·업무 채팅·자유보고
- 복잡한 Google API·OAuth·Drive 권한 자동연동
- 자동 게시·고급 홍보 분석
- 영상 자동분석과 대용량 스트리밍
- 복잡한 다단계 승인과 모든 행동 로그
- 법정 급여·근태 자동계산

## 10. 기획 종료 판단

현재 기준으로 다음은 모두 확보되었다.

- 사업 목적과 기능 채택 기준
- 조직·역할·대시보드 원칙
- 일반 장애인 근로자 화면
- 현장 운영·반복 템플릿·최소기록
- 업무유형별 기록 수준
- 근로자지원·홍보·승인·협업 원칙
- 계정·퇴사·보안의 기본원칙
- 파일·Google 링크의 초기 범위
- 지원사업·접근성 확장 방향

따라서 **새로운 주요 기능 논의는 중단하고, 남은 미정사항은 구현 전 운영결정 또는 기술설계 항목으로 이동할 수 있다.**
