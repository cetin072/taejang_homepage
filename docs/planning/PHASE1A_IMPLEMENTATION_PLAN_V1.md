# 태장 업무플랫폼 Phase 1A 구현계획 v1

상태: **확정**

## 1. 목표

Phase 1A는 로그인·가입승인·조직·역할·일반 장애인 근로자 정보게시판·관리자 작성화면·공용 태블릿·기본 감사기록을 실제로 사용할 수 있는 수준까지 구현한다.

다만 첫 개발 PR은 아래 `1A-0` 조사와 `1A-1` 보안 스파이크로 제한한다. `1A-2`부터 `1A-8`은 보안 검증을 통과한 뒤 별도 PR에서 순차 진행하며, 첫 작업에 정보게시판 전체 화면·현장관리·상담·홍보·Google API·AI 기능을 포함하지 않는다.

공개 홈페이지의 정적 HTML/CSS/JavaScript 구조와 장애 경계를 유지하고, 내부 업무플랫폼 장애가 공개 홈페이지 장애로 이어지지 않게 한다.

## 2. 개발 원칙

- 새 개발 브랜치와 Draft PR에서 작업한다.
- PR #17의 기획문서는 먼저 기준본으로 정리한다.
- 각 작업 묶음은 독립적으로 실행·테스트 가능해야 한다.
- 인증과 RLS를 화면 구현보다 먼저 검증한다.
- 일반 근로자 화면은 읽기·확인 중심이며 업무입력을 추가하지 않는다.
- 임시로 RLS를 끄거나 service role 키를 브라우저에 넣는 방식은 금지한다.
- 기존 공개 홈페이지·Netlify 배포·콘텐츠 허브 기능을 회귀시키지 않는다.

## 3. 작업 묶음

### 1A-0. 저장소·기술구조 재확인

목표:

- 기존 관리자 프로토타입과 Supabase 관련 문서·코드 조사
- 공개 홈페이지와 내부 플랫폼 라우팅·빌드·배포 경계 확인
- Phase 1A 구현 위치 결정

산출물:

- 현재 구조 조사 보고
- 구현 디렉터리·라우팅 제안
- 사용 라이브러리·빌드방식 결정
- 공개 홈페이지 무변경 기준

완료조건:

- 임의 기술교체 없이 기존 기준에 맞는 구조가 선택됨
- 필요한 환경변수 목록이 문서화됨

### 1A-1. Supabase 보안 스파이크

목표:

- Auth 가입
- 승인 대기 프로필 생성
- active 상태 확인
- 역할·범위 조회
- 정지·퇴사 세션차단
- 마지막 최고관리자 보호

산출물:

- 최소 migration
- RLS 정책
- 보안 테스트
- 실패·위험 보고

필수 검증:

- pending 사용자의 내부 데이터 접근 차단
- 다른 사용자·부서 데이터 접근 차단
- departed 전환 후 기존 세션 차단
- 마지막 super_admin 변경 거부

이 단계가 통과하지 않으면 UI 개발을 확대하지 않는다.

### 1A-2. 계정·조직·역할 관리

목표:

- 회원가입
- 승인 대기 화면
- 최고관리자 가입 승인
- 부서·직책·역할·작업반 배정
- 계정 상태 관리

화면:

- 로그인
- 회원가입
- 승인 대기
- 가입 승인 목록
- 사용자 상세
- 조직·역할 관리
- 계정 상태 변경

완료조건:

- 가입만으로 내부정보를 볼 수 없음
- 김형철 계정을 초기 최고관리자로 지정할 안전한 절차가 있음
- 일반 직원은 계정상태와 역할을 변경할 수 없음
- 감사로그가 남음

### 1A-3. 작업방법 관리

목표:

- 작업방법 기본정보 작성
- 3~7단계 사진·쉬운 문장 작성
- 미리보기
- editing/active/retired 상태

화면:

- 작업방법 목록
- 작성·수정
- 단계 편집
- 모바일·태블릿 미리보기
- 일반 근로자 단계별 보기

완료조건:

- active 전 필수항목과 단계수 검증
- 일반 근로자는 active 자료만 읽기 가능
- retired 자료는 과거 기록으로 유지

### 1A-4. 오늘의 업무·일정·공지·안내 관리

목표:

- 관리자 작성
- 개인·작업반·부서 대상 지정
- 표시기간·상태 처리
- 쉬운 문장 중심 미리보기

기능:

- 오늘의 업무 CRUD와 취소·보관
- 일정 CRUD와 변경·취소
- 공지 CRUD와 중요·확인필요
- 자주 보는 안내 CRUD와 정렬

완료조건:

- 대상 범위가 정확히 적용됨
- 개인 지정이 작업반·부서 기본값보다 우선함
- 긴 문장 경고·필수항목 누락 안내가 작동함

### 1A-5. 일반 장애인 근로자 정보게시판

목표:

- 로그인 후 바로 필요한 정보 확인
- 지속적인 휴대전화 사용을 유도하지 않는 흐름

화면:

- 오늘
- 오늘의 업무 상세
- 작업방법 목록·단계별 상세
- 내 일정
- 중요공지
- 자주 보는 안내

완료조건:

- 시작·완료·수량·실적·출퇴근 입력이 없음
- 큰 글자·큰 버튼·쉬운 문장·사진 중심
- 본인·소속 대상 정보만 조회
- 중요한 공지만 확인처리 가능

### 1A-6. 공용 태블릿·안내화면

목표:

- 작업반 단위의 오늘 업무·공지·작업방법 제공
- 개인 로그인 없이도 제한된 현장안내 가능

완료조건:

- 개인 일정·개인업무·상담·인사정보 노출 없음
- 기기 토큰 원문을 저장하지 않음
- 최고관리자가 기기를 비활성화할 수 있음
- 화면잠금·재접속 상황을 고려함

### 1A-7. 감사·운영 안전장치

목표:

- 중요 변경 추적
- 상태변경·역할변경·게시행위 기록

필수 로그:

- 가입 승인·거절
- 역할·부서·작업반 변경
- 정지·퇴사·복구·삭제
- 마지막 최고관리자 보호 동작
- 작업방법 게시·사용중지
- 중요공지 게시·취소
- 공용기기 생성·비활성화

완료조건:

- 일반 사용자가 로그를 수정·삭제할 수 없음
- 민감 본문·토큰·비밀번호가 로그에 없음

### 1A-8. 통합검수와 현장 샘플

샘플 데이터:

- 생산 오전반·오후반
- 물류 오전반·오후반
- 일반 근로자 2~4명
- 반장·팀장·최고관리자 계정
- 작업방법 2개
- 오늘 업무·일정·중요공지·자주 보는 안내
- 공용 태블릿 1대

검수:

- 역할별 화면 노출
- 대상범위
- 접근성
- 모바일·PC·태블릿 반응형
- 계정 정지·퇴사
- RLS 우회 시도
- 공개 홈페이지 회귀

## 4. PR 분할 권장

Phase 1A 전체를 한 PR에 몰아넣지 않는다.

권장 순서:

1. 보안 스파이크 PR
2. 계정·조직 기반 PR
3. 작업방법 관리 PR
4. 업무·일정·공지 관리 PR
5. 일반 근로자 정보게시판 PR
6. 공용 태블릿·감사·통합검수 PR

다만 초기 기술구조에 따라 2~3개 묶음으로 통합할 수 있으며, 각 PR은 적용 기획문서와 검수표를 명시한다.
