3. 계정상태 기준
==================================================

필수 상태:

- `pending`: 승인 대기
- `active`: 사용 중
- `suspended`: 일시 정지
- `departed`: 퇴사
- `deleted`: 삭제 처리

필수 동작:

- 회원가입 직후 `pending`
- `pending`은 로그인 여부와 무관하게 내부자료 조회·작성·수정 불가
- `active`만 배정된 범위에서 접근 가능
- `suspended`, `departed`, `deleted`는 내부 접근 차단
- 퇴사 처리 시 기존 세션이 남아 있어도 서버·DB 계층에서 차단
- 과거 작성물·승인·감사로그의 작성자 참조는 보존
- 일반 사용자는 자신의 상태를 바꿀 수 없음

==================================================
4. 최소 데이터 구조
==================================================

문서 기준을 따르되 실제 저장소와 Supabase 권장방식에 맞춰 migration을 작성하세요.

최소 후보:

- `profiles`
- `departments`
- `positions`
- `roles`
- `profile_roles`
- 필요 시 `profile_departments` 또는 프로필의 부서 참조
- `account_status_history`
- `audit_logs`

직책은 표시용이며 권한의 유일한 근거가 아닙니다. 실제 접근은 계정상태·역할·부서 또는 담당범위로 판단해야 합니다.

==================================================
5. 시스템 최고관리자
==================================================

초기 시스템 최고관리자는 김형철입니다.

다음을 구현하거나 안전한 초기화 절차로 문서화하세요.

- 초기 최고관리자 지정 방법
- 일반 브라우저 코드에서 임의 승격 불가
- 최고관리자 권한 변경은 감사로그 기록
- 마지막 활성 최고관리자의 정지·퇴사·삭제·권한회수 차단
- 본인이 마지막 최고관리자인 경우 자기 권한 제거 차단

이메일 주소나 Auth UUID가 저장소에 확정되어 있지 않다면 임의로 만들지 말고, 안전한 bootstrap 절차와 필요한 운영 입력값을 문서화하세요.

==================================================
6. RLS·서버 권한 기준
==================================================

반드시 DB 또는 신뢰 가능한 서버 계층에서 강제하세요.

검증 항목:

- `pending` 사용자의 select/insert/update/delete 차단
- `suspended`, `departed`, `deleted` 사용자의 모든 내부 데이터 접근 차단
- 일반 사용자의 profiles 상태·역할 변경 차단
- 일반 사용자의 audit log 작성 위조·수정·삭제 차단
- 최고관리자 전용 변경 경로
- 서비스 역할 키 브라우저 노출 금지
- RLS가 꺼진 테이블이 생기지 않도록 검사

UI에서 메뉴나 버튼을 숨기는 것만으로 권한을 구현하지 마세요.

==================================================
7. 세션 차단
==================================================

정지·퇴사 처리 직후 기존 로그인 세션의 접근도 차단해야 합니다.

다음을 조사하고 구현 또는 검증하세요.

- Supabase Auth의 세션·토큰 특성
- DB RLS에서 매 요청 현재 profile 상태 확인
- 필요 시 서버 측 세션 종료 또는 사용자 로그아웃 처리
- 이미 발급된 토큰으로 내부 데이터 접근이 가능한지 테스트

완전한 즉시 토큰 폐기가 기술적으로 제한된다면, 그 한계와 보완책을 명확히 보고하고 DB 계층 차단은 반드시 구현하세요.
