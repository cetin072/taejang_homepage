# 태장 업무플랫폼 Phase 1A 접근권한·RLS 기준 v1

상태: **확정**

## 1. 목적

이 문서는 Phase 1A의 계정 승인·역할·부서·작업반·일반 근로자 정보게시판·관리자 작성화면·공용 태블릿 데이터에 적용할 접근권한과 Supabase RLS 원칙을 정한다.

## 2. 공통 원칙

- 모든 업무 테이블은 기본적으로 RLS를 활성화한다.
- 화면에서 버튼을 숨기는 것만으로 권한을 구현하지 않는다.
- `pending`, `suspended`, `departed`, `deleted` 계정은 업무 데이터 접근을 차단한다.
- 역할은 신뢰 가능한 서버 측 조회 또는 검증된 claim을 사용한다.
- 이름·이메일 문자열로 권한을 판정하지 않는다.
- 시스템 최고관리자 김형철도 실제 계정 생성 후 `super_admin` 역할로 배정한다.
- 서비스 역할 키는 서버 전용으로만 사용하고 브라우저에 노출하지 않는다.
- 대표이사·운영총괄의 넓은 조회권한에도 민감정보 분리와 감사기록을 적용한다.

## 3. 인증 상태별 접근

### 비로그인

허용:

- 로그인
- 회원가입
- 비밀번호 재설정

차단:

- 모든 내부 업무 데이터
- 공용기기 전용 토큰 화면을 제외한 내부 API

### `pending`

허용:

- 본인 승인 대기 상태
- 본인 최소 프로필
- 로그아웃

차단:

- 업무·일정·공지·작업방법
- 다른 사용자·조직·역할
- 관리자 기능

### `active`

역할과 범위에 따라 접근한다.

### `suspended`, `departed`, `deleted`

- 신규 세션 발급 차단
- 기존 세션 무효화
- 모든 업무 테이블 select/insert/update/delete 차단
- 과거 작성자·수정자 참조는 유지

## 4. 역할별 핵심 정책

### 일반 장애인 근로자 `general_worker`

조회 가능:

- 본인 프로필의 비민감 표시정보
- 본인 또는 소속 작업반·부서를 대상으로 한 오늘의 업무
- 본인 또는 소속 범위의 일정
- 본인 또는 소속 범위의 활성 공지
- 본인 업무에 연결된 활성 작업방법
- 본인 또는 소속 범위의 자주 보는 안내
- 본인의 공지 확인기록

작성 가능:

- 본인의 `notice_confirmations` 생성

작성 불가:

- 업무·일정·공지·작업방법 수정
- 시작·완료·실적·출퇴근 기록
- 다른 직원 정보 조회

### 현장 반장·관리자 `field_lead`

Phase 1A 조회:

- 본인 부서·작업반의 사용자 최소정보
- 본인 범위의 업무·일정·공지·작업방법

Phase 1A 작성:

- 권한을 부여받은 경우 본인 범위의 업무·공지·작업방법 작성·수정

차단:

- 상담 원문
- 민감 인사정보
- 전사 계정상태 변경

### 부서 팀장 `department_lead`

- 본인 부서의 업무·일정·공지·작업방법 관리
- 본인 부서 사용자 최소정보 조회
- 다른 부서 데이터는 부서 간 공유 대상으로 지정된 항목만 조회
- 계정 상태·최고관리자 역할 변경 불가

### 운영총괄 `operations_manager`

- 전 부서 업무·일정·공지·작업방법 조회
- 중요 운영 데이터 작성·수정·승인
- 부서·작업반 운영관계 조회·조정
- 대표이사 보고 대상 확인
- 계정 역할 부여는 정책상 허용할 수 있으나 Phase 1A에서는 최고관리자 화면을 통해 수행하고 감사로그를 남긴다.

### 대표이사 `ceo`

- 전사 업무 데이터 조회
- 중요 경영·최종승인 대상 조회·판단
- 일상 계정관리 UI는 기본 대시보드에서 제외
- 필요 시 세부조회는 가능하되 변경행위는 역할에 따라 제한하고 감사기록을 남긴다.

### 시스템 최고관리자 `super_admin`

- 가입 승인·거절
- 부서·직책·역할·작업반 배정
- 계정 정지·퇴사·복구·삭제 처리
- 조직 기준정보 관리
- 공용기기 관리
- 감사기록 조회

보호 규칙:

- 마지막 활성 `super_admin`의 역할 회수·정지·퇴사·삭제 금지
- 자기 자신의 최고관리자 역할 회수는 다른 활성 최고관리자가 있을 때만 허용
- 최고관리자 변경은 항상 감사로그와 사유를 남김

## 5. 대상 범위 판정

업무·일정·공지·안내는 다음 대상 필드를 사용할 수 있다.

- `profile_id`: 개인 대상
- `work_group_id`: 작업반 대상
- `department_id`: 부서 대상
- 대상 필드가 모두 비어 있고 전사 공개 플래그가 있는 경우 전체 활성 사용자 대상

일반 근로자 조회 조건은 다음 중 하나다.

- `profile_id = current_profile_id`
- `work_group_id`가 현재 활성 작업반 소속과 일치
- `department_id`가 현재 부서와 일치
- 전사 대상이며 상태·표시기간 조건을 충족

개인 대상과 작업반·부서 대상이 충돌하면 개인 대상 설정을 우선한다.

## 6. 테이블별 RLS 요약

### `profiles`

- 본인: 본인 비민감 필드 조회
- 팀장·반장: 본인 범위의 최소 표시필드 조회
- 운영총괄·대표이사: 전사 조회
- 최고관리자: 전사 조회·승인·상태·조직배정 수정
- 일반 사용자의 `account_status`, 역할, 승인정보 직접 수정 금지

민감 필드는 공개용 view 또는 안전한 RPC로 분리한다.

### `departments`, `positions`, `roles`

- 활성 사용자: 필요한 기준정보 조회
- 최고관리자: 생성·수정·비활성화
- 사용 중인 항목의 물리삭제 금지

### `profile_roles`, `work_group_members`

- 본인: 본인 배정 조회
- 팀장·반장: 본인 범위 조회
- 운영총괄·대표이사: 전사 조회
- 최고관리자: 부여·회수·변경

### `worker_tasks`

- 일반 근로자: 본인 대상범위와 활성 상태만 조회
- 팀장·반장: 본인 범위 조회·작성·수정
- 운영총괄: 전사 조회·작성·수정
- 삭제 대신 취소·보관 상태 사용

### `work_guides`, `work_guide_steps`

- 일반 근로자: `active` 자료만 조회
- 작성권한자: 본인 부서 또는 담당 자료 작성·수정
- 중요 안전·품질 자료의 활성화는 팀장 또는 운영총괄 권한으로 제한
- `retired` 자료는 일반 근로자에게 노출하지 않고 관리자 기록으로 보존

### `schedules`

- 일반 근로자: 본인 대상범위의 일정 조회
- 팀장·반장: 본인 범위 작성·수정
- 운영총괄: 전사 관리

### `notices`

- 일반 근로자: 본인 대상범위·활성기간·활성상태 공지만 조회
- 관리자 작성권한자: 역할 범위 내 작성·수정
- 중요공지의 게시·취소는 팀장 이상 권한

### `notice_confirmations`

- 일반 근로자: 본인 행만 조회·생성
- 본인 확인기록 수정·삭제 금지
- 팀장·반장: 본인 범위 공지의 확인 여부 조회
- 확인기록은 평가·생산성 데이터와 결합하지 않는다.

### `frequent_guides`

- 일반 근로자: 본인 대상범위의 활성 안내 조회
- 팀장·관리자: 역할 범위 내 작성·수정

### `shared_devices`

- 일반 사용자 직접 조회 금지
- 최고관리자만 생성·비활성화
- 공용기기는 원문 토큰이 아닌 해시 검증 또는 서버 발급 세션 사용
- 개인 대상 데이터와 프로필 상세 조회를 차단

### `account_status_history`, `audit_logs`

- 일반 사용자 접근 금지
- 운영총괄·대표이사: 업무상 필요한 범위 조회
- 최고관리자: 전사 조회
- 일반 update/delete 금지
- 감사로그에 비밀번호·토큰·민감 본문 전체 저장 금지

## 7. 안전한 View·RPC 권장

직접 테이블 조회보다 다음 안전 계층을 우선 검토한다.

- `current_user_context`: 본인 상태·부서·직책·역할·작업반
- `worker_today_board`: 본인에게 허용된 오늘 업무·일정·공지 집계
- `shared_group_board`: 공용기기용 개인식별 제거 데이터
- `admin_user_directory`: 관리자 범위에 맞춘 최소 사용자 정보
- `approve_pending_user(...)`: 가입승인·배정·감사로그를 트랜잭션으로 처리
- `change_account_status(...)`: 상태변경·세션 차단·이력·재배정 검증
- `set_profile_roles(...)`: 최고관리자 보호와 감사기록을 포함한 역할 변경

중요 상태변경을 여러 클라이언트 쿼리로 나누지 않고 서버 함수 또는 Edge Function에서 원자적으로 처리한다.

## 8. 세션과 퇴사 처리

계정상태 변경 시 애플리케이션 RLS만 믿지 않는다.

- Auth 세션 만료·강제 로그아웃 처리
- 모든 RLS 정책에서 `active` 상태를 재확인
- write RPC에서도 `active` 상태를 재검증
- 담당업무 재배정 전까지 퇴사자에게 새 업무를 지정할 수 없도록 제한
- 과거 작성물·감사로그·공지 작성자 관계 유지

## 9. 파일과 이미지

Phase 1A에서 이미지 저장소를 사용할 경우:

- 일반 근로자는 승인된 작업방법·공지 이미지 읽기만 가능
- 작성자는 본인 역할범위의 임시경로에 업로드
- 활성 자료에서 참조 중인 파일은 직접 삭제 금지
- 공용기기에는 개인사진·상담자료·인사파일 전달 금지
- 버킷도 공개 홈페이지용과 내부 업무플랫폼용을 분리

Google Drive 링크는 URL 메타데이터만 저장하며 Drive 권한을 플랫폼이 대신 보장하지 않는다.

## 10. Phase 1A 보안 테스트 필수항목

- 비로그인 사용자가 내부 API를 호출할 수 없는가
- `pending` 사용자가 URL을 직접 입력해도 데이터가 보이지 않는가
- 일반 근로자가 다른 직원의 개인업무·일정을 조회할 수 없는가
- 다른 부서 팀장이 URL 또는 API 조작으로 타 부서 자료를 수정할 수 없는가
- 공용기기에서 개인 대상 업무와 개인정보가 노출되지 않는가
- 정지·퇴사 처리 직후 기존 세션으로 읽기·쓰기가 차단되는가
- 마지막 최고관리자 역할 회수·정지·퇴사가 거부되는가
- 감사로그를 일반 사용자가 수정·삭제할 수 없는가
- retired 작업방법과 만료 공지가 일반 화면에서 제외되는가
- 공지 확인을 다른 사용자 명의로 생성할 수 없는가

## 11. 구현 전 확인사항

- Supabase custom claim 사용 여부와 DB 역할 조회 방식 결정
- Auth 가입 후 `profiles` 생성 트리거의 최소 필드 확정
- 세션 강제무효화 구현 방식 검증
- 공용기기 인증을 별도 익명 세션·단기 토큰·서버 프록시 중 하나로 결정
- 이미지 Storage 버킷과 경로 규칙 확정

이 항목은 개발 PR의 Phase 1A 보안 스파이크에서 실제 동작을 검증한다.
