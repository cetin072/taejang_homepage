# PR #17 최종 정리·개발 인계 기준 v1

상태: **확정**

## 1. 목적

PR #17을 태장 업무플랫폼의 기획·기술설계 기준본으로 마무리하고, 이후 실제 구현은 별도의 개발 브랜치와 Draft PR에서 진행한다.

## 2. PR #17의 역할

PR #17에는 다음만 포함한다.

- 프로젝트 헌장
- 조직·역할·보고체계
- MVP 최종 기능명세
- 역할별 권한표
- 화면 목록과 이동도
- Phase 1A 데이터 모델
- Phase 1A 접근권한·RLS 기준
- Phase 1A 구현계획
- 개발 작업 인계 기준

공개 홈페이지 코드, 관리자 프로토타입 코드, Supabase migration, 실제 로그인 기능, 배포 설정은 PR #17에서 변경하지 않는다.

## 3. 병합 전 필수 확인

1. PR이 Draft 상태인지 확인한다.
2. `main` 최신 상태를 가져온다.
3. `mergeable: false` 원인이 문서 충돌인지 확인한다.
4. 충돌이 있으면 기획 브랜치에서 최신 `main`을 반영하고 문서 의미가 바뀌지 않도록 해결한다.
5. 공개 홈페이지 HTML/CSS/JavaScript, Netlify, Supabase 설정이 변경되지 않았는지 재확인한다.
6. 문서 링크가 깨지지 않는지 확인한다.
7. `docs/README.md`에서 최종 기준문서가 모두 연결되는지 확인한다.
8. 새 기능이 기획 종료 승인 이후 임의로 추가되지 않았는지 확인한다.
9. 사용자 최종 승인 후에만 Ready for review 또는 병합 단계로 전환한다.
10. 병합 방식은 저장소 기준에 따라 squash merge를 우선 검토한다.

## 4. 병합 후 개발 브랜치 원칙

PR #17 병합 후 최신 `main`에서 새 개발 브랜치를 만든다.

권장 브랜치명:

`codex/admin-phase1a-security-foundation`

권장 Draft PR 제목:

`feat: establish phase 1a auth and security foundation`

기획 브랜치와 개발 브랜치를 재사용하지 않는다. `main`에 직접 수정·push·merge하지 않는다.

## 5. 첫 개발 작업 범위

첫 개발은 전체 업무플랫폼 구현이 아니라 보안 기반 검증으로 제한한다.

포함:

- 현재 저장소·관리자 프로토타입 구조 조사
- Supabase Auth 적용 위치 결정
- `pending`, `active`, `suspended`, `departed`, `deleted` 계정상태
- 승인 대기 사용자의 내부자료 접근 차단
- 프로필·부서·직책·역할 최소 데이터 구조
- 시스템 최고관리자 김형철 초기 지정 전략
- 마지막 최고관리자 보호
- 정지·퇴사자의 새 요청과 기존 세션 차단
- 역할·상태 기반 RLS 최소 정책
- 감사로그 최소 구조
- 자동화 가능한 보안 테스트
- 운영·배포 문서 보완

제외:

- 일반 근로자 정보게시판 전체 UI
- 작업방법 편집기
- 오늘의 업무·일정·공지 관리
- 반복 업무 템플릿
- 현장 기록
- 근로자지원·상담·홍보 모듈
- Google Workspace API 연동
- AI 기능

## 6. 첫 개발 작업의 완료 조건

- 승인 대기 계정은 인증 후에도 내부 데이터 조회·작성 불가
- 활성 계정만 허용된 범위의 데이터 접근 가능
- 정지·퇴사 전환 후 새 요청 차단
- 기존 세션 무효화 또는 이에 준하는 서버 측 차단 검증
- 일반 사용자가 계정상태·역할을 변경할 수 없음
- 마지막 최고관리자 제거·정지·퇴사 방지
- RLS 우회가 불가능한 테스트 확보
- 비밀값이 저장소에 포함되지 않음
- 공개 홈페이지 기능 회귀 없음
- 변경·테스트·잔여 위험이 Draft PR에 기록됨

## 7. 작업 결과 검토 순서

1. 코덱스가 조사 결과와 구현방향을 먼저 보고한다.
2. 기존 문서와 충돌이 있으면 코딩 전에 중단하고 보고한다.
3. 보안 기반을 구현하고 테스트한다.
4. Draft PR과 커밋을 만든다.
5. 기획방에서 결과를 검토한다.
6. 통과한 경우에만 Phase 1A 계정·조직 관리 화면으로 확장한다.

## 8. 금지사항

- PR #17에서 구현 코드를 함께 개발하지 않는다.
- 첫 개발 작업에서 정보게시판·현장관리·전문업무까지 범위를 넓히지 않는다.
- 화면에서 버튼을 숨기는 것만으로 권한을 구현하지 않는다.
- 서비스 역할 키나 관리자 비밀값을 브라우저에 노출하지 않는다.
- 일반 사용자가 자신의 승인상태·역할·부서·직책을 변경할 수 있게 하지 않는다.
- 정지·퇴사자를 단순 UI 리다이렉트만으로 차단하지 않는다.
- 사용자 승인 없이 `main`에 병합하지 않는다.
