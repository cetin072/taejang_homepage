# 태장 업무플랫폼 계정·직책·역할 배정 기준 v1

상태: **확정**

## 1. 기본 원칙

태장 업무플랫폼은 실제 담당자 이름을 개발 단계에서 고정하지 않는다. 계정이 생성된 뒤 시스템 최고관리자가 가입 상태와 실제 조직 배치를 확인해 부서·직책·역할·권한을 배정한다.

개발 기준은 개인 이름보다 역할과 권한 구조를 우선한다.

## 2. 가입 후 배정 흐름

1. 직원 또는 관계자가 계정을 생성한다.
2. 계정은 `pending` 승인 대기 상태로 등록한다.
3. 시스템 최고관리자가 가입자 신원과 실제 소속을 확인한다.
4. 부서, 직책, 역할, 근무반, 화면 지원방식을 배정한다.
5. 필요한 권한만 활성화한다.
6. 직책이나 부서가 변경되면 같은 계정의 배정정보를 수정한다.

가입만으로 내부 업무정보나 민감정보에 자동 접근할 수 없어야 한다.

## 3. 직책 기준

초기 플랫폼은 다음 직책·역할을 이름과 분리해 관리한다.

- 대표이사
- 운영총괄
- 사내이사
- 시스템 최고관리자
- 부서 팀장
- 총괄반장
- 업무반장
- 현장 관리자
- 근로자지원 책임자
- 근로자지원 담당자
- 근로지원인
- 일반 직원
- 일반 장애인 근로자
- 외부 지도자

한 사람이 여러 역할을 겸임할 수 있지만, 각 권한은 명시적으로 배정한다.

## 4. 시스템 최고관리자

시스템 최고관리자는 **김형철**로 확정한다.

시스템 최고관리자는 다음을 담당한다.

- 가입 계정 확인과 승인
- 부서·직책·역할·권한 배정
- 계정 상태 변경
- 정지·퇴사·삭제 처리
- 담당 업무와 콘텐츠 재배정
- 마지막 최고관리자 보호
- 중요 권한 변경 기록 확인

대표이사 역할과 시스템 최고관리자 역할은 구분한다. 대표이사는 회사의 최종 의사결정자이며, 시스템 최고관리자는 플랫폼 계정과 운영권한을 관리한다.

## 5. 계정 상태

계정 상태는 다음을 사용한다.

- pending: 회원가입 후 승인 대기. 본인 최소정보와 승인 대기 안내 외 내부자료 접근 불가
- active
- suspended
- departed
- deleted

`pending`은 운영 중인 활성·정지·퇴사 상태와 구분되는 가입 승인 전 상태다. 최고관리자가 신원·소속을 확인하고 부서·직책·역할·근무반을 배정한 뒤에만 `active`로 전환한다.

초기 최고관리자 지정은 일반 회원가입 화면이나 담당자 이름 하드코딩으로 처리하지 않는다. 김형철의 실제 Auth 계정 식별값을 운영자가 안전한 초기화 절차에 입력하고, 지정 결과와 이후 최고관리자 변경을 감사기록에 남긴다.

## 6. 권한 변경 원칙

- 직책 변경 시 새 계정을 만들지 않고 기존 계정의 역할을 수정한다.
- 일반 직원은 다른 직원의 계정 상태나 직책을 변경할 수 없다.
- 현장 반장에게 상담 원문이나 민감 인사정보 권한을 부여하지 않는다.
- 역할을 숨기는 화면처리만으로 권한을 구현하지 않고 데이터 계층에서도 제한한다.
- 최고관리자 변경·삭제·정지는 중요 변경으로 기록한다.
- 시스템에는 최소 한 명의 활성 최고관리자가 남아 있어야 한다.

## 7. 개발 검수 기준

- 가입 직후 자동으로 내부 권한이 부여되지 않는가
- `pending` 계정이 내부자료를 조회·작성·수정하지 못하는가
- 최고관리자가 가입자를 확인한 뒤 직책과 권한을 배정할 수 있는가
- 담당자 이름을 코드에 고정하지 않고 역할로 관리하는가
- 김형철이 초기 시스템 최고관리자로 설정되는가
- 대표이사와 시스템 최고관리자 권한이 개념적으로 분리되는가
- 부서·직책 변경 시 계정을 유지할 수 있는가
- 마지막 최고관리자 보호와 중요 변경 기록이 적용되는가
