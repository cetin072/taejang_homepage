# 자주 보는 안내 구현

상태: **개발 검증용 — Draft PR 승인 전 운영 Supabase에 적용하지 않는다.**

## 1. 기능 목적과 중요공지의 구분

일반 근로자가 회사생활·안전·지원 정보를 필요할 때 쉽게 다시 찾는 읽기 전용 안내 기능이다. 근태, 상담기록, 평가, 확인률 또는 업무 실적 기능이 아니다.

중요공지는 현재 바뀐 일정·장소·안전사항처럼 게시기간, 중요도·긴급도, 필요한 경우 확인 흐름을 갖는 소식이다. 자주 보는 안내는 평소 차량 탑승장소, 휴게시간, 화장실·비상구 위치처럼 장기간 참고하는 고정 정보다. 동일 내용을 양쪽에 중복 등록하지 않는다.

## 2. 분류·대상 범위

분류는 근무시간, 휴게·식사, 장소안내, 안전, 복장·준비물, 지각·결근, 급여·서류, 도움요청, 회사생활, 기타로 고정한다.

기존 `company | department | work_group | profile` 대상 범위를 재사용한다. 활성 일반 근로자는 회사 전체, 본인 부서, 현재 유효 작업반, 본인 개인 대상만 본다. 다른 부서·작업반·개인, 초안, 사용 중지, 적용 전·종료 자료는 목록과 상세 모두 차단한다.

## 3. 데이터 모델·RLS·RPC

`20260724000100_frequent_staff_guidance.sql`은 `staff_guidance_items`를 추가한다. 제목, 쉬운 요약·본문, 장소, 승인된 표시명의 도움요청 방법, 관련 작업방법·일정, 안전한 HTTPS 링크, 대상, 표시순서, 상단 고정, 상태, 적용기간, 생성·수정자·시각, 변경 사유를 기록한다. 첨부파일과 Storage는 범위 밖이다.

직접 테이블 권한은 회수하고 RLS를 활성화한다. 일반 근로자는 `get_my_staff_guidance_list`, `get_my_staff_guidance_detail`만 사용한다. 관리자는 `list_manageable_staff_guidance`, `get_staff_guidance_for_edit`, `save_staff_guidance`을 사용한다. 모든 security-definer 함수는 `search_path`를 고정하고 활성계정·기존 대상·새 대상을 서버에서 재검증한다.

## 4. 일반 근로자 화면

홈의 다섯 번째 핵심 메뉴인 `자주 보는 안내`가 실제 목록으로 연결된다. 목록은 큰 분류 버튼, 제목·요약 검색, 상단 고정→표시순서→최근 수정일→제목 정렬, 빈 상태 `등록된 안내가 없습니다.`를 제공한다. 상세는 분류, 쉬운 요약·본문, 적용 대상 설명, 장소, 도움요청 방법, 관련 작업방법·일정, 안전한 외부 링크, 등록일·수정일만 빈 항목 없이 표시한다.

안전은 `안전 관련 안내`, 도움요청은 `도움이 필요할 때 사용하세요`, 상단 고정은 `꼭 기억해 주세요`를 색상과 함께 글자로 표시한다. 확인 버튼·읽음 평가·근태 기록은 없다.

일반 근로자는 모바일 우선 단일 열을 유지하며 PC에서도 기존 600px 이하 고정폭을 따른다. 44px 이상 버튼, 키보드 포커스, `aria-pressed` 필터, 로딩·빈 상태·오류 분리, 쉬운 문장, reduced motion 기존 기준을 적용한다.

## 5. 관리자 관리·권한

관리자는 작성, 임시저장, 게시, 수정, 사용 중지, 대상·분류·표시순서·상단고정·적용기간 설정과 일반 근로자 미리보기를 사용한다. 물리삭제는 제공하지 않는다.

- `super_admin`, `operations_manager`: 전사와 모든 대상 범위
- `department_lead`: 본인 부서, 그 부서 작업반·개인
- `field_lead`: 담당 작업반과 그 반원 개인
- 일반 직원·일반 근로자: 작성·수정·게시·사용 중지 불가

관리자 셸에 `안내 관리`를 추가한다. 권한 없는 역할은 메뉴를 보지 못하고, 직접 URL·RPC 호출도 기존 guarded RPC가 차단한다.

## 6. 링크·개인정보·감사로그

관련 링크는 HTTPS만 허용하며 사용자정보, 제어문자, `javascript:`, `data:`를 거부한다. 실제 주소를 보이고 새 창에는 `noopener noreferrer`을 사용하며 사용자 입력 HTML은 삽입하지 않는다.

개인 상담기록, 장애유형·질병명, 주민등록번호, 계좌번호, 개인 전화번호 목록, 평가·징계·근태 상세, 비밀번호·토큰은 입력하지 않는다. 문의 담당자는 승인된 표시명과 회사 연락방법만 기록한다.

생성, 게시, 수정, 사용 중지, 대상·분류·표시순서·상단고정·적용기간·관련 링크·작업방법 변경을 기존 `audit_logs`에 상태·변경 여부 중심으로 기록한다. 장문 본문, 개인 연락처, 토큰, 민감정보는 metadata에 남기지 않는다.

## 7. 검증·제외·후속

정적 테스트와 전용 pgTAP는 테이블·RLS·RPC 권한·안전 링크·화면 모듈을 검증한다. GitHub Actions는 깨끗한 migration, DB lint, 전체 pgTAP, 기존 Auth/Data API·Today·작업방법·일정·공지 회귀를 계속 실행한다.

상담 사례관리, 근태·출퇴근, 평가·읽음률, 댓글·채팅, 파일·이미지 업로드와 Storage, 통합검색, PWA·푸시, Google Workspace API, 운영 Supabase·실제 직원 데이터는 이번 범위에서 제외한다. 후속에서는 실제 비운영 환경에서 화면 문구·대상 범위·도움 요청 담당자 표시 기준을 검수한 뒤, 필요 시 안전한 첨부·사진 참조를 별도 설계한다.
