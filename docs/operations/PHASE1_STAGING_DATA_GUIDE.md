# Phase 1 비운영 샘플 데이터 가이드

## 원칙

- 운영 Supabase, 실제 직원 이름·이메일·전화번호·장애·건강·상담·인사 정보는 사용하지 않는다.
- migration에는 샘플 데이터를 넣지 않는다.
- 계정은 `example.test` 가상 정보만 사용하고 고정 비밀번호를 문서·저장소에 남기지 않는다.
- service role 키는 사용하지 않는다. publishable key와 가상 계정 세션의 guarded RPC만 사용한다.

| 예시 표시명 | 역할 | 부서·작업반 |
| --- | --- | --- |
| 검수 최고관리자 | `super_admin` | 운영 |
| 검수 대표이사 | `ceo` | 운영 |
| 검수 운영총괄 | `operations_manager` | 운영 |
| 검수 생산팀장 | `department_lead` | 생산 |
| 검수 A반 책임자 | `field_lead` | 생산 / A반 |
| 검수 사무담당자 | `office_staff` | 운영 |
| 검수 근로자 A·B | `general_worker` | 생산 / A반, 물류 / B반 |
| 검수 빈 상태 근로자 | `general_worker` | 물류 / B반 |

생산·물류 2개 부서와 A·B반 2개 작업반을 만든다. 시작일, 종료된 멤버십, 오늘 시작 멤버십을 하나씩 두어 KST 경계를 확인한다.

콘텐츠는 오늘 업무, 3단계 이상 작업방법, 회사·부서·작업반·개인 대상 일정·공지·안내, 취소 일정, 수정 공지, 적용 전·종료 안내, 타 범위·초안·사용 중지 자료와 빈 상태를 각각 준비한다. 실제 연락처·개인 사례·건강 정보는 넣지 않는다.

1. 운영과 구분된 테스트 프로젝트 또는 로컬 Supabase에 clean migration을 적용한다.
2. 가상 계정을 가입·bootstrap·승인하고 guarded RPC 또는 관리자 UI로 자료를 입력한다.
3. 수동 체크리스트와 Auth/Data API 검증을 수행한다.
4. 종료 후 테스트 프로젝트를 폐기하거나 로컬 DB는 `supabase db reset`으로 초기화한다. 운영 DB에서 샘플 삭제를 시험하지 않는다.
