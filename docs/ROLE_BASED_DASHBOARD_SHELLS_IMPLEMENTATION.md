# 역할별 대시보드·공통 앱 셸 구현

상태: **구현 완료 (화면 골격)**

## 1. 목적과 범위

PR #23의 Today·일정·공지·작업방법을 역할별 첫 화면에서 안전하게 찾을 수 있게 한다. 생산·물류·영업·결재·근로자지원 사례관리의 완성형 기능은 이번 범위가 아니다.

기존 `get_my_access_context()`의 계정상태·역할 우선순위와 모든 RPC/RLS 판정을 그대로 사용한다. URL의 `role` 또는 `home` 값은 역할을 결정하지 않으며, `/app/` 진입·새로고침·뒤로가기는 항상 서버 접근 문맥을 다시 확인한다.

## 2. 공통 앱 셸

일반 근로자 외 역할은 좌측 사이드바, 상단 헤더, 현재 역할·사용자 표시, 로그아웃, 본문 대시보드로 구성한다. 모바일에서는 44px 이상의 메뉴 버튼으로 사이드바를 열고 닫으며 `aria-expanded`, `aria-controls`, 현재 메뉴 `aria-current`을 제공한다.

관리 메뉴는 `super_admin`, `operations_manager`, `department_lead`, `field_lead`에게만 표시한다. 메뉴 표시는 편의 기능일 뿐이며, 일정·공지·Today·작업방법의 실제 조회와 변경은 기존 guarded RPC와 RLS가 재검증한다. 일반 근로자는 관리자 모듈을 동적으로 불러오지 않는다.

## 3. 역할별 홈

| 역할 | 홈 원칙 | 실제 연결 |
| --- | --- | --- |
| 시스템 최고관리자 | 운영 요약에 계정 승인 바로가기 추가 | 실제 `list_pending_profiles`와 기존 보호된 승인 화면 |
| 대표이사 | 중요공지·주요 일정 중심의 가장 간결한 요약 | 본인 대상 안전 RPC만 사용, 전사 상세·통계는 준비 중 |
| 운영총괄 | 오늘 처리할 일, 일정, 중요공지, 작업방법, 빠른 이동 | 현재 안전 RPC 요약, 근로자지원 특이사항은 준비 중 |
| 부서 팀장 | 우리 부서에 적용되는 일정·공지와 관리 자료 | 현재 사용자에게 적용되는 부서 대상 안전 RPC |
| 현장 책임자 | 오늘 작업·장소, 안전공지, 변경된 작업방법 중심 | 현재 사용자·작업반 대상 안전 RPC와 관리 화면 |
| 전문 담당자·일반 사무직 | 일정·공지·작업방법만 우선 표시 | 현재 구현된 자료만 표시, 홍보·지원 상세 기능은 준비 중 |
| 일반 근로자 | Today·작업방법·내 일정·중요공지 | 기존 모바일 정보게시판 유지 |

중복 역할은 `super_admin → ceo → operations_manager → department/professional → field_lead → general staff → general_worker`의 기존 `ROLE_ROUTES` 우선순위를 유지한다.

## 4. 일반 근로자 화면

일반 근로자는 좌측 사이드바와 관리자형 대시보드를 사용하지 않는다. 360px부터 단일 열, 큰 글자와 44px 이상 터치영역을 유지하며 PC에서도 콘텐츠 폭을 600px 이하로 제한한다. Today, 작업방법, 내 일정, 중요공지의 기존 정보 순서와 읽기 전용 원칙을 유지한다.

## 5. 실제 데이터와 준비 중 처리

요약 카드는 `get_my_schedule_list`, `get_my_notice_list`, `get_my_work_guide_list`와 최고관리자 전용 `list_pending_profiles`의 실제 응답만 사용한다. 데이터가 없으면 숫자 대신 `현재 ... 없습니다`로 안내한다.

생산·물류·영업·결재, 근로자지원 상세 사례관리, 전사 경영통계, 통합검색은 가짜 메뉴·통계·동작 버튼을 만들지 않는다. 화면에 필요할 때만 `준비 중` 카드로 명확히 표시한다.

## 6. 디자인·접근성

공통 토큰은 배경, 카드, 글자, 보조글자, 테두리, 브랜드, 성공·주의·위험·정보, 그림자, 반경, 콘텐츠 폭을 정의한다. 상태는 색상과 문구를 함께 제공하며 로딩·빈 상태·오류는 구분한다. 키보드 포커스, landmark, 제목 계층, reduced motion을 고려한다.

## 7. Google Workspace 향후 결합

이번 구현은 Google Workspace API, OAuth, Gmail, Calendar 동기화, Drive 연결을 만들지 않는다. 후속 모듈에서 실제 권한·운영 승인 후 공용 Calendar, Drive 문서, Meet, 회사 이메일을 각 역할의 빠른 이동 위치에 연결할 수 있다. 태장 업무플랫폼 DB는 계속 직원 화면의 기준 데이터이며 Google 도구는 보조 수단이다.

## 8. 후속 개발 순서

1. 실제 운영 역할·부서·작업반으로 각 요약 범위 검수
2. 현장 실행 기록과 부서 업무 모듈을 별도 설계·검증
3. 근로자지원 민감정보 분리 모듈 설계
4. 홍보 모듈 및 승인 흐름 구현
5. 검증된 모듈에 한해 Google Workspace 보조 연결 검토

운영 Supabase 적용, 실제 계정·직원 데이터 입력, 서비스 역할 키 사용은 이번 범위에서 제외한다.
