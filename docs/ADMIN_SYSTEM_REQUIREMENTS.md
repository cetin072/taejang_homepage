# 직원용 관리자 시스템 요구사항

## 1. 목적과 범위

비개발자 홍보팀 2~4명이 코드·GitHub·배포 설정을 직접 다루지 않고 콘텐츠를 작성하고, 검토·승인 후 안전하게 공개하는 시스템을 목표로 합니다.

### 직원이 관리하는 영역

- 공지·일터 이야기·활동 글 작성과 임시저장
- 승인된 사진·PDF 첨부, 미리보기, 검토 요청
- 승인·반려·예약 게시·게시 취소·결과 확인
- 검색·휴지통 복원·수정 이력·간단 보고

### 개발자만 관리하는 영역

- 디자인·레이아웃·메뉴·HTML/CSS/JavaScript
- DB 스키마·RLS·보안 정책·환경 변수
- Netlify·Supabase·도메인·배포·장애 대응

### 범위

- 1차 MVP: 로그인, 공지·일터·활동, 사진, 임시저장, 검토·승인·게시, 휴지통, 기본 로그
- 후속: 보도자료·채용·PDF·배너·팝업·예약 게시·미디어 고도화·보고서
- 이번 설계 제외: 실제 관리자 UI, Supabase 프로젝트, SQL migration, 로그인·DB·Storage·Netlify 연결

## 2. 사용자 역할

| 권한 | 작성자 | 검토자 | 게시 담당 | 최고관리자 | 개발자 |
| --- | --- | --- | --- | --- | --- |

| 작성 | 가능 | 가능 | 가능 | 가능 | 테스트 콘텐츠만 |
| 수정 | 본인 초안·수정 요청 건 | 검토 의견, 직접 본문 수정은 제한 | 승인 후 공개 설정만 | 전체 | 구조 수정만 |
| 승인·반려 | 불가 | 가능 | 불가 | 가능 | 불가 |
| 게시·게시 취소 | 불가 | 불가 | 가능 | 가능 | 긴급 복구만 |
| 삭제 | 본인 미게시 초안의 휴지통 이동 요청 | 불가 | 게시 취소 후 휴지통 이동 | 휴지통 이동 가능 | 불가 |

| 계정 관리 | 불가 | 불가 | 불가 | 발급·역할 변경·일시 정지·퇴사 처리·복구·재배정 | 기술 복구 지원, 인사 판단 불가 |
| 설정 관리 | 불가 | 불가 | 공개일 등 제한 설정 | 업무 설정 일부 | 코드·DB·보안·배포 설정 |

한 사람이 여러 역할을 겸할 수 있지만 `작성 → 최종 공개`는 최소 두 사람이 확인하는 것을 기본으로 합니다. 최고관리자도 중요한 외부 공개는 검토 기록을 남깁니다.

### 계정 상태

| 상태 | 의미 | 기본 처리 |
| --- | --- | --- |
| `active` | 정상 사용 | 역할 범위 안에서 로그인·조회·변경 허용 |
| `suspended` | 장기 휴직·조사·보안 사고 등에 따른 일시 정지 | 즉시 세션을 종료하고 모든 관리자 변경 작업 차단 |
| `departed` | 퇴사 또는 업무 종료 처리 | 즉시 접근을 차단하되 과거 작성물과 이력 보존 |
| `deleted` | 예외적인 법적·운영상 영구삭제 절차 완료 | 로그인·조회·변경 차단, 보존 의무 확인 후에만 사용 |

일반 퇴사는 `departed`, 일시적 차단은 `suspended`를 사용합니다. `deleted`는 일반 퇴사에 사용하지 않으며 Phase 2 이후 별도 승인·보존 절차로 검토합니다. 작성자·검토자·게시 담당은 본인 또는 타인의 계정 상태를 변경할 수 없습니다. 최고관리자만 일시 정지·퇴사 처리·복구와 담당자 재배정을 할 수 있고, 마지막 활성 최고관리자는 자기 자신을 비활성화하거나 역할을 낮출 수 없습니다. 개발자는 장애·보안 정책의 기술 복구만 담당하며 퇴사 여부는 최고관리자 또는 회사 책임자가 승인합니다.

### 비활성 계정 차단 범위

`suspended` 또는 `departed` 전환 즉시 새 로그인, 기존 세션, refresh token 재발급, 관리자 화면, 콘텐츠 작성·수정·검토 요청·승인·게시, 사진·파일 업로드, 미디어 교체, Storage 삭제, 관리자 API 변경 요청을 차단합니다. 버튼을 숨기는 것으로 끝내지 않고 Supabase Auth, 관리자 애플리케이션 서버 또는 API, DB Row Level Security, Storage 정책에서 모두 `profiles.status = active`를 확인합니다.

최고관리자의 상태 변경 요청은 재확인 후 처리하고 Auth 계정 차단 또는 ban, 세션·refresh token 무효화 방식을 검토합니다. 관리자 API는 요청마다 활성 상태를 확인하므로 이미 페이지를 열어 둔 기기도 다음 저장·업로드 요청이 거부되고 로그인 화면으로 이동해야 합니다. 구체적인 Supabase Auth 회수 방식은 Phase 0에서 공식 문서를 확인한 뒤 확정합니다. 상태 변경자·시각·이전 상태·새 상태·사유·처리 결과는 감사 로그에 남깁니다.

### 기존 작성물 보존과 담당자 재배정

- 게시 글, 초안, 검토·승인·반려 기록, 업로드 사진, 첨부파일, revision history, audit log는 계정 상태와 무관하게 보존합니다.
- 공개 홈페이지에는 퇴사 상태를 표시하지 않습니다. 관리자 화면에는 작성자 이름 옆에 `퇴사` 상태를 표시할 수 있습니다.
- 작성자 ID와 표시명은 감사 목적으로 유지하며, 비활성 계정을 무명 또는 삭제된 사용자로 바꾸지 않습니다.
- 작성 중 초안, 수정 요청·검토 요청·승인 대기 콘텐츠, 예약 게시물, 내부 담당 업무, 미처리 업로드, 보도자료·채용공고 초안과 향후 미처리 문의는 활성 직원에게 재배정할 수 있어야 합니다.
- 재배정은 현재 담당자만 바꾸고 원 작성자 정보는 revision에 유지합니다. 변경자·변경 시각·사유를 기록합니다.
- 예약 게시물은 담당자 미지정 상태로 두지 않으며 재배정 실패 항목이 있으면 퇴사 확정 전에 경고합니다.

### 계정 발급·회수 절차

입사·업무 배치: `최고관리자 계정 발급 → 역할 지정 → 초기 비밀번호 또는 초대 링크 발송 → 최초 로그인 → 비밀번호 변경 → 관리자 사용 안내 확인 → 활성화 기록` 순서로 처리합니다.

퇴사: `업무 종료 확인 → 작성 중 콘텐츠 확인 → 예약 게시물 확인 → 담당자 재배정 → departed 처리 → 기존 세션 종료 → 로그인·DB·Storage 차단 확인 → 회사 이메일·외부 계정 회수 확인 → 처리자·처리 시각 기록 → 24시간 이내 재점검` 순서로 처리합니다.

긴급 차단: `suspended 처리 → 즉시 세션 종료 → 작성·업로드 차단 → 최고관리자 확인 → 조사 후 active 또는 departed 전환` 순서로 처리합니다.

## 4. 콘텐츠 상태

| 상태 | 의미 |
| --- | --- |
| `draft` | 작성 중 |
| `review_requested` | 검토 요청 |
| `changes_requested` | 수정 요청 |
| `approved` | 공개 내용 승인 |
| `scheduled` | 예약 게시 대기 |
| `published` | 게시 중 |
| `archived` | 공개 종료·보관 |
| `trashed` | 휴지통, soft delete |

### 상태 전환과 권한

| 현재 → 다음 | 허용 역할 | 필수 조건 |
| --- | --- | --- |
| `draft` → `review_requested` | 작성자 | 필수값·자동검증 통과 |
| `review_requested` → `changes_requested` | 검토자 | 수정 사유 기록 |
| `review_requested` → `approved` | 검토자 | 개인정보·사실·승인 확인 |
| `changes_requested` → `review_requested` | 작성자 | 수정 내용 기록 |
| `approved` → `published` | 게시 담당 | Preview·공개일·최종 승인 확인 |
| `approved` → `scheduled` | 게시 담당 | 예약 시각·시간대 확인 |
| `scheduled` → `published` | 시스템·게시 담당 | 발행 성공 기록 |
| `published` → `archived` | 게시 담당·최고관리자 | 종료 사유 기록 |
| `archived` → `published` | 게시 담당·최고관리자 | 재게시 승인 |
| 모든 상태 → `trashed` | 최고관리자, 제한된 게시 담당 | 게시 중이면 먼저 공개 처리 결정 |
| `trashed` → 이전 상태 | 최고관리자 | 복원 사유·이전 상태 기록 |

승인 이후 본문·사진이 바뀌면 승인을 무효화하고 다시 `review_requested`로 보내는 것이 기본입니다.

## 6. 콘텐츠 편집 화면

### 입력

- 콘텐츠 유형·카테고리·제목·요약·도입문
- 중간 제목과 본문 블록
- 대표 사진·갤러리·사진 설명·대체 텍스트
- 첨부파일·공개 희망일·내부 메모
- 기업명·로고 승인, 인물 공개 동의, 개인정보 확인

### 기능

- 자동저장과 명시적 임시저장
- 실제 홈페이지 형태의 미리보기
- 검토 요청과 변경사항 경고
- 필수값·ID·카테고리·날짜 검사
- 중복 ID 방지와 안전한 ID 자동 제안
- 본문 블록 순서 이동
- 승인 후 변경 시 재승인 안내
- Phase 2의 예약 게시

직원에게 HTML 입력란을 제공하지 않습니다. 본문은 제목·문단·목록·인용처럼 허용된 구조화 블록만 사용하고, 위험한 태그는 저장·출력 단계에서 차단합니다.

## 7. 미디어 관리

- 업로드 시 안전한 내부 파일명으로 정규화하고 원본 표시명은 메타데이터로 보관합니다.
- 원본, 웹 최적화본, 썸네일을 구분합니다.
- JPG·PNG·WebP·승인된 SVG만 허용하고 실행 파일·HTML·스크립트형 SVG는 차단합니다.
- 파일 크기·픽셀 크기·가로세로 비율을 검사하고 웹용 변환을 수행합니다.
- alt, 출처, 촬영일, 업로더, 승인 상태, 인물·로고 승인, 사용 콘텐츠를 기록합니다.
- 중복 해시와 유사 이미지 후보를 표시하되 자동 삭제하지 않습니다.
- 공개 중인 콘텐츠가 참조하는 파일은 일반 직원이 삭제할 수 없습니다.
- 삭제는 휴지통으로 이동하고 원본·파생본·참조 관계를 함께 복원합니다.

## 8. 첨부파일 관리

대상은 PDF 회사소개서·보도자료·채용 파일·인증 공개본입니다.

- 확장자뿐 아니라 MIME·파일 시그니처·용량을 검사합니다.
- 악성 파일 검사는 서버 측 검사 또는 검증 서비스 도입을 검토합니다.
- 공개본과 내부 검토본을 분리하고 내부 파일은 공개 URL을 발급하지 않습니다.
- 문서 유형·버전·발행일·승인자·현재 대표 파일·대체 파일을 기록합니다.
- 이전 버전은 비공개 보관하며, 공개 URL 유지·종료 정책을 유형별로 결정합니다.

## 9. 검토·승인 흐름

`직원 작성 → 내부 검토 → 수정 요청 또는 승인 → 게시 담당 확인 → 게시 → 결과 기록`

- 검토 의견과 수정 요청 사유는 콘텐츠 revision에 연결합니다.
- 승인자·승인 시각·승인 대상 revision을 고정합니다.
- 게시 후 긴급 수정은 사유와 범위를 기록하고 재승인 여부를 결정합니다.
- 게시 취소·승인 취소 기준을 운영 정책으로 정하고 감사 로그를 남깁니다.

## 10. 미리보기

- 실제 홈페이지 CSS·구조와 같은 렌더링을 사용합니다.
- PC·390px·360px 전환을 제공합니다.
- 비공개·만료형 Preview URL 또는 인증된 관리자 Preview를 사용합니다.
- 승인 전 콘텐츠가 검색엔진·공개 API·공개 Storage에 노출되지 않도록 합니다.
- 최종 승인자는 Preview revision과 게시 revision이 같은지 확인합니다.

## 11. 검색·필터

제목, 작성자, 상태, 유형, 카테고리, 작성·공개 기간, 공개 여부, 사진 유무, 승인 대기, 수정 요청 여부를 조합해 검색합니다. 기본 정렬은 최근 수정순이며 게시 목록은 공개일 최신순을 지원합니다.

## 12. 삭제·휴지통·복원

- 일반 직원에게 영구삭제 권한을 주지 않습니다.
- 삭제는 `trashed` 상태와 삭제 시각·삭제자·이전 상태를 기록하는 soft delete입니다.
- 보존 기간은 사용자 결정 후 설정하며 기간 만료 전 최고관리자 확인을 요구합니다.
- 게시 URL은 삭제 후 404, 보관 안내, 대체 콘텐츠 연결 중 유형별 정책을 정합니다.
- 첨부와 미디어는 다른 콘텐츠 참조 여부를 확인한 뒤 처리합니다.
- 영구삭제 후에도 필요한 감사 기록은 별도 보존 정책에 따라 최소 정보만 유지합니다.

## 13. 변경 이력과 감사 로그

로그 대상은 로그인, 작성, 수정, 검토 요청, 수정 요청, 승인, 반려, 게시, 게시 취소, 삭제, 복원, 계정·권한 변경입니다.

| 항목 | 기록 기준 |
| --- | --- |
| 사용자·시각 | 내부 사용자 ID와 서버 시각 |
| 대상 | 콘텐츠·미디어·계정 ID와 종류 |
| 이전·변경 값 | 민감정보를 제외한 변경 diff 또는 revision 연결 |
| 사유 | 승인·반려·긴급 수정·삭제에 필수 |
| IP·세션 | 보안 목적의 최소 범위, 보존 기간과 접근 권한 제한 |

비밀번호·토큰·민감한 원문은 로그에 기록하지 않습니다.

## 19. 기술 구조 선택

| 방식 | 장점 | 단점·위험 | 비용·난이도 | 전환 비용 |
| --- | --- | --- | --- | --- |
| A. 현재 정적 사이트 + 관리자 수동 반영 | 공개 사이트 안정, 초기 단순 | 관리자 입력 후 개발자 수동 반영, 지연·중복 | 낮은 비용·낮은 초기 난이도 | 낮음 |
| B. 공개 페이지가 Supabase 직접 조회 | 게시 즉시 반영, 배포 감소 | DB·네트워크 장애가 공개 페이지에 영향, 공개 권한 설계 중요 | 런타임 비용·중간 난이도 | 중간~높음 |
| C. Supabase에서 승인 콘텐츠를 정적 JSON으로 발행 | 공개 사이트 안정, 관리자와 공개 코드 분리, revision·Preview 가능 | 발행 파이프라인 필요, 게시마다 제한적 배포 발생 | 중간 난이도, 배포를 묶어 크레딧 관리 가능 | 중간 |

**최종 권장안은 C입니다.** 관리자 시스템은 Supabase에서 작성·검토·승인을 처리하고, 게시 시 승인 revision만 정적 JSON으로 발행합니다. 공개 홈페이지는 같은 도메인의 정적 JSON을 읽거나 빌드 결과에 포함하므로 관리자 장애가 기존 공개 화면을 중단시키지 않습니다. 게시 건을 묶거나 예약 발행해 Netlify 빌드 횟수를 관리합니다.

## 21. Storage 설계

| 버킷 | 용도 | 공개성 |

## 22. 공개 홈페이지 연동 설계

| 후보 | 공개 안정성 | 게시 속도 | 운영 부담 | 판단 |
| --- | --- | --- | --- | --- |
| Supabase 직접 조회 | DB 장애 영향 큼 | 즉시 | 공개 RLS·캐시 필요 | 초기 비권장 |
| 빌드 시 Supabase→JSON | 높음 | 배포 완료 후 | 빌드 연결 필요 | 권장 기반 |
| 게시 시 정적 JSON 생성·배포 | 높음 | 수분 내 | 발행·실패 복구 필요 | **1차 권장** |
| Netlify Function 조회 | 함수 장애·비용 영향 | 즉시 | 함수·캐시·보안 필요 | 후속 검토 |

1차는 승인된 콘텐츠 snapshot을 정적 JSON으로 생성하고 Draft Preview에서 확인한 뒤 배포하는 방식입니다. 현재 `content.js` 구조를 JSON 어댑터로 옮겨 페이지 렌더링 변경을 최소화합니다.

각 Phase는 기능 완료뿐 아니라 직원 매뉴얼, 권한 회귀, 모바일 확인, 백업·복구 시연, 사용자 승인까지 완료 기준에 포함합니다.

### Phase 1 계정 생명주기 테스트

- `active` 직원의 로그인·초안 작성·사진 업로드가 역할 범위에서 성공합니다.
- `suspended` 전환 즉시 새 로그인, 기존 세션 저장, 업로드가 실패하고 과거 작성물은 유지됩니다.
- `departed` 전환 즉시 로그인·세션 재사용·작성·수정·검토 요청·사진·파일 업로드가 실패합니다.
- 퇴사자의 게시물·revision·audit log는 유지되고 관리자 내부에만 퇴사 상태가 표시됩니다.
- 최고관리자가 `active`로 복구할 때 역할을 재확인하고 기존 세션을 되살리지 않은 채 새 로그인을 요구합니다.
- 일반 직원·검토자·게시 담당의 계정 상태 변경과 마지막 활성 최고관리자 비활성화가 실패합니다.
- 초안·예약 게시물 재배정이 성공하고 원 작성자 revision은 유지되며 비활성 계정을 새 담당자로 지정할 수 없습니다.
- 모든 계정 상태 변경과 재배정에 감사 로그가 생성됩니다.
