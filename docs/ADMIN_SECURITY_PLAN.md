# 관리자 시스템 보안 계획

이 문서는 설계 기준이며 실제 키·토큰·비밀번호·접속정보를 포함하지 않습니다.

## 1. 인증

- 관리형 인증(Supabase Auth 우선 검토)을 사용하고 자체 비밀번호 저장을 금지합니다.
- 초기 로그인은 업무용 이메일과 비밀번호를 권장하며 소셜 로그인은 운영 필요 확인 후 도입합니다.
- 이메일 확인, 비밀번호 최소 기준, 재설정 토큰 만료, 반복 로그인 제한을 적용합니다.
- 최고관리자·개발자 다중요소 인증은 Phase 1 또는 직후 우선 도입합니다.

## 2. 권한

- 작성자·검토자·게시 담당·최고관리자·개발자의 최소 권한을 분리합니다.
- UI 버튼 숨김은 편의 기능일 뿐이며 모든 읽기·쓰기 권한은 서버와 DB에서 다시 검사합니다.
- 승인자와 게시자는 승인된 revision만 처리할 수 있습니다.
- 개발자라도 업무상 필요 없는 초안·개인정보를 상시 열람하지 않습니다.
- 모든 역할 권한보다 `profiles.status = active` 검사를 먼저 적용합니다. 작성자·검토자·게시 담당은 계정 상태를 변경할 수 없고, 최고관리자만 일시 정지·퇴사 처리·복구·재배정을 수행합니다.
- 개발자는 Auth·DB·보안 정책의 기술 복구를 지원하지만 퇴사 여부를 임의로 판단하지 않으며 최고관리자 또는 회사 책임자의 승인을 따릅니다.

## 3. Row Level Security

- 모든 관리자 테이블은 기본 거부 후 필요한 행위만 허용합니다.
- 작성자는 본인 초안과 수정 요청 건, 검토자는 검토 대상, 게시 담당은 승인·예약·게시 대상을 봅니다.
- 최고관리자 권한은 일반 브라우저 service key가 아니라 검증된 사용자 역할로 제한합니다.
- 공개 사용자는 발행 allowlist를 통과한 공개 view 또는 정적 JSON만 읽습니다.
- RLS 정책은 역할별 허용·거부 자동 테스트를 갖춥니다.
- `active` 계정만 역할 범위 안에서 INSERT·UPDATE·DELETE할 수 있습니다. `suspended`와 `departed`는 변경 작업을 모두 거부하고 `deleted`는 로그인·관리자 조회·변경을 모두 거부합니다.
- 본인 작성물 정책이나 역할 정책보다 계정 활성 상태 검사를 먼저 적용합니다.
- 비활성 계정과 상태 변경 이력의 감사 목적 조회는 최고관리자와 필요한 개발자에게만 제한합니다.

## 4. Storage 정책

| 영역 | 읽기 | 쓰기 | 핵심 제한 |
| --- | --- | --- | --- |
| `public-media` | 공개 | 승인된 서버 발행 절차 | 승인본·웹 최적화본만 |
| `private-originals` | 제한 역할 | 작성자 이상 업로드 | 원본 공개 URL 금지 |
| `public-files` | 공개 | 승인된 발행 절차 | PDF 공개본만 |
| `private-review-files` | 검토 관련 역할 | 작성자 이상 | 만료형 접근, 검색 노출 금지 |

파일 확장자·MIME·시그니처·용량을 검사하고 실행 가능한 HTML·스크립트·위험한 SVG를 차단합니다. 공개 파일명에 개인 이름·전화번호·내부 프로젝트명을 사용하지 않습니다.

- `active` 계정만 역할상 허용된 버킷에 업로드·교체·삭제할 수 있습니다.
- `suspended`와 `departed` 계정의 업로드·교체·삭제를 모두 차단하되 기존 파일은 자동 삭제하지 않습니다.
- 비공개 원본 읽기도 역할과 활성 상태를 함께 검사합니다.
- 공개 파일 URL의 읽기 정책과 관리자 변경 권한을 분리해 공개 파일이 있다는 이유로 수정·삭제가 허용되지 않게 합니다.

## 5. 공개 키와 비밀 키

- 공개용 anon key와 서버 전용 service key를 구분합니다.
- service key, GitHub 토큰, Netlify build hook, 이메일 비밀값을 브라우저 번들·GitHub·문서·로그에 넣지 않습니다.
- 비밀값은 승인된 환경 변수 저장소에 두고 역할·환경별로 분리·회전합니다.
- 공개 키도 RLS를 우회할 수 없도록 정책을 검증합니다.

## 6. 환경 변수

- 개발·Preview·운영 값을 분리합니다.
- 환경 변수 이름과 목적만 문서화하고 값은 문서화하지 않습니다.
- 로그·오류 화면·클라이언트 소스맵에 비밀값이 포함되지 않는지 검사합니다.
- 퇴사·사고·외부 협력 종료 시 관련 비밀값 회전 절차를 둡니다.

## 7. 계정 발급·회수

- 최고관리자가 업무용 이메일로 초대하고 역할·만료·발급 사유를 기록합니다.
- 공용 계정 사용을 금지하고 1인 1계정을 원칙으로 합니다.
- 일반 퇴사는 `departed`, 장기 휴직·조사·보안 사고의 임시 차단은 `suspended`로 처리합니다. 예외적 영구삭제 완료 상태인 `deleted`는 일반 퇴사에 사용하지 않습니다.
- 퇴사·휴직·직무 변경 시 Auth·API·RLS·Storage 접근을 즉시 차단하고 기존 세션과 refresh token을 회수합니다.
- 마지막 활성 최고관리자 계정 중지와 자기 역할 강등·자기 비활성화를 방지합니다.
- 최고관리자 계정 상태 변경은 재확인을 요구하고 `audit_logs`와 계정 상태 이력에 반드시 기록합니다.
- 분기별 활성 계정·역할 검토를 권장합니다.

### 입사·업무 배치

1. 최고관리자가 계정을 발급합니다.
2. 역할을 지정합니다.
3. 초기 비밀번호 또는 초대 링크를 발송합니다.
4. 직원이 최초 로그인합니다.
5. 비밀번호를 변경합니다.
6. 관리자 사용 안내를 확인합니다.
7. 활성화 기록을 남깁니다.

### 퇴사

1. 퇴사 또는 업무 종료를 확인합니다.
2. 작성 중 콘텐츠를 확인합니다.
3. 예약 게시물을 확인합니다.
4. 초안·검토·예약·미처리 업무를 활성 직원에게 재배정합니다.
5. 계정을 `departed`로 전환합니다.
6. 기존 세션과 refresh token을 종료합니다.
7. 로그인·DB·Storage 접근 차단을 확인합니다.
8. 회사 이메일과 외부 업무 계정 회수를 확인합니다.
9. 처리자·처리 시각·최소한의 업무상 사유를 기록합니다.
10. 24시간 이내 차단과 재배정 상태를 다시 확인합니다.

### 긴급 차단

1. 계정을 `suspended`로 전환합니다.
2. 기존 세션을 즉시 종료합니다.
3. 작성·업로드·관리자 API 변경을 차단합니다.
4. 최고관리자가 영향을 확인합니다.
5. 조사 후 `active` 복구 또는 `departed` 전환을 승인합니다.

## 8. 비밀번호 재설정과 세션

- 재설정 링크는 1회용·단기 만료로 하고 성공 후 기존 세션 종료를 검토합니다.
- 관리자 세션은 적절한 유휴·절대 만료를 설정하고 최고관리자 작업은 재인증합니다.
- 쿠키 사용 시 Secure·HttpOnly·SameSite 정책을 적용합니다.
- 계정 일시 정지·퇴사 처리에는 모든 기기의 세션·refresh token 강제 종료를 Phase 1 필수 기능으로 둡니다.
- 관리자 API는 요청마다 `profiles.status = active`를 검사해 이미 열린 페이지의 저장·업로드 요청도 거부합니다. 비활성화 후 다음 요청에서는 세션을 정리하고 로그인 화면으로 이동시킵니다.
- 복구는 기존 세션을 되살리지 않고 역할 확인 후 새 로그인을 요구합니다.
- Supabase Auth의 ban·세션 무효화 방식은 Phase 0에서 공식 문서를 확인한 뒤 구현 방식을 확정합니다.

## 9. 감사 로그

- 로그인 성공·실패, 콘텐츠·미디어·계정·역할·게시 행위를 기록합니다.
- 계정 일시 정지·퇴사 처리·복구·담당자 재배정에는 이전 상태, 새 상태, 처리자, 시각, 사유, 결과를 기록합니다.
- 사용자, 서버 시각, 대상, 행위, 사유, 결과, 제한된 세션 참조를 기록합니다.
- 비밀번호·토큰·민감 문서 원문은 기록하지 않습니다.
- 보존 기간과 조회 역할을 결정하고 로그 자체 변경·삭제를 제한합니다.

## 10. 업로드 보안

- 허용 형식·최대 용량·최대 픽셀을 서버에서 검사합니다.
- 파일명은 안전한 내부 키로 바꾸고 경로 이동 문자를 제거합니다.
- 이미지 재인코딩으로 불필요한 메타데이터 제거를 검토합니다.
- PDF·첨부는 악성 파일 검사 완료 전 공개하지 않습니다.
- 얼굴·기업명·로고·송장·주소·건강정보는 자동검사만 믿지 않고 사람이 확인합니다.

## 11. 개인정보

- 업무에 필요한 최소 직원 계정 정보만 저장합니다.
- 주민등록번호, 근로자 장애·건강정보, 개인 연락처, 고객사 계약 원문은 관리자 콘텐츠 DB에 저장하지 않습니다.
- 인물 동의서 원본은 별도 제한 보관소에 두고 DB에는 확인 상태·확인자·일자·참조번호만 둡니다.
- 개인정보 처리 목적·보유기간·파기·권리 행사를 실제 운영에 맞게 확정합니다.

## 12. 백업과 복구

- DB 자동 백업, 정기 논리 내보내기, Storage 목록·중요 원본 백업을 분리합니다.
- 백업도 운영 DB와 같은 수준으로 암호화·접근 통제합니다.
- 최소 분기별로 콘텐츠·미디어·사용자 권한 복구를 시험하고 결과를 기록합니다.
- 마지막 정상 정적 발행본을 유지해 관리자 장애 중에도 공개 홈페이지를 제공합니다.

## 13. 사고 대응

1. 계정·키·업로드·게시 사고 범위를 확인합니다.
2. 관련 계정·세션·키·발행을 제한합니다.
3. 마지막 승인 정적본으로 공개 화면을 유지하거나 복구합니다.
4. 감사 로그와 변경 revision으로 원인을 확인합니다.
5. 사용자·관계자 통지 필요성을 담당자가 판단합니다.
6. 복구 후 권한·키·정책을 재검증하고 사고 기록을 남깁니다.

비밀값 유출이 의심되면 값을 문서나 이슈에 복사하지 않고 즉시 회전합니다.

## 14. 환경 분리

| 환경 | 데이터 | 접근 | 배포 |
| --- | --- | --- | --- |
| 개발 | 가상·비민감 테스트 데이터 | 개발자 | 로컬 또는 개발 전용 |
| Preview | 검토용 복제·비공개 데이터 | 인증 사용자·만료 링크 | Draft PR Preview |
| 운영 | 승인된 실제 콘텐츠 | 역할별 최소 권한 | 승인 후 `main` 발행 |

Preview가 운영 Storage 원본이나 운영 service key를 직접 사용하지 않도록 합니다. 환경별 DB·버킷·키 분리 수준은 비용과 위험을 Phase 0에서 결정합니다.

## 15. 보안 완료 기준

- 역할별 허용·거부 RLS 테스트 통과
- 비밀 키의 저장소·브라우저·로그 노출 0건
- 업로드 위험 형식·과대 파일 차단
- 승인 없는 공개·게시·삭제·계정 변경 차단
- 계정 중지와 세션 회수 시나리오 통과
- `suspended`·`departed`의 Auth·API·RLS·Storage 변경 차단과 과거 작성물 보존 확인
- 마지막 활성 최고관리자 보호, 계정 복구 시 새 로그인, 담당자 재배정·감사 로그 확인
- 백업 복구 시연과 감사 로그 확인
