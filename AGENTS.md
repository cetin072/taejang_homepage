# 태장 홈페이지 작업 기준

이 저장소는 농업회사법인 태장 주식회사의 공식 홈페이지와 태장 업무 플랫폼을 함께 발전시키는 프로젝트입니다.

## 최상위 기준

- 모든 기획, 개발, 콘텐츠, 운영 작업 전에 [`PROJECT_CHARTER.md`](PROJECT_CHARTER.md)를 먼저 읽습니다.
- 홈페이지와 업무 플랫폼은 태장의 장애인 고용 안정, 장기 완전고용 유지, 제품·서비스 경쟁력, 수익성, 영업과 사회공헌을 지원하는 도구입니다.
- 기능 개발 자체를 목표로 삼지 않습니다.
- 새 기능은 프로젝트 헌장의 기능 채택 판단 기준을 통과해야 합니다.
- 전체 시스템을 한 번에 만들지 않고 모듈별로 실제 운영 가능한 수준까지 완성한 뒤 다음 모듈로 확장합니다.

## 현재 기술 구조

- 공개 홈페이지는 순수 HTML, CSS, JavaScript로 구성된 정적 사이트입니다.
- 공통 스타일은 `assets/css/styles.css`, 공통 동작은 `assets/js/site.js`에 있습니다.
- 활동·일터 목록 콘텐츠는 `assets/js/content.js`와 `assets/js/listing.js`를 사용합니다.
- Netlify는 저장소 루트를 정적 파일로 배포합니다.
- 내부 업무 플랫폼은 공개 홈페이지와 연결할 수 있지만 보안, 권한과 장애 경계는 분리합니다.

## 브랜치와 검토 원칙

- `main` 브랜치에 직접 수정하거나 직접 커밋하지 않습니다.
- 모든 변경은 목적이 드러나는 기능 브랜치에서 작업합니다.
- 변경 후 검토 가능한 설명과 테스트 결과를 남기고 PR로 `main` 반영을 요청합니다.
- PR 검토와 사용자 승인이 끝나기 전에는 `main` 병합, 운영 배포, Netlify 설정 변경을 하지 않습니다.

## 수정·보안 원칙

- 요청 범위 밖의 파일은 수정하지 않습니다.
- 사용자 승인 없이 운영 URL, 도메인, Netlify 환경 변수, 배포 설정을 변경하지 않습니다.
- 비밀값은 저장소에 넣지 않습니다.
- 민감정보, 주주명부, 주민등록번호, 계약서 원문, 초상권 사용 여부가 확인되지 않은 사진은 공개 저장소에 올리지 않습니다.
- 건강·장애·인사 등 민감정보는 필요 최소한으로 수집하고 일반 업무 데이터와 분리합니다.
- 화면에서 버튼을 숨기는 것만으로 권한을 구현하지 않고, 데이터 계층에서도 접근을 제한합니다.

## 상시 운영 기준

- 모든 작업 전에 `PROJECT_CHARTER.md`와 `docs/operations/` 문서를 확인합니다.
- 파일은 `PROJECT_STRUCTURE.md`와 `ASSET_POLICY.md`에 지정된 위치에만 저장합니다.
- 반복 작업은 `COMMAND_LIBRARY.md`의 템플릿을 우선 사용합니다.
- 브랜치·검사·PR 절차는 `CODEX_WORKFLOW.md`를 따릅니다.
- 새 라이브러리나 도구 도입 전 기존 구조로 해결 가능한지 확인합니다.
- 사용하지 않는 자산·코드·중복 구현을 남기지 않습니다.
- 성능 최적화가 디자인과 기능 품질을 훼손하지 않도록 균형을 유지합니다.
- 기존 로컬 미커밋 변경을 사용자 승인 없이 새 작업에 포함하지 않습니다.
- 파일을 받았다고 무조건 저장소에 등록하지 않고 `ASSET_POLICY.md` 기준으로 판정합니다.

## 콘텐츠와 대외 표현

- 콘텐츠·문구 작업 전에는 [`docs/reference/TAEJANG_PUBLIC_WEB_BRIEF.md`](docs/reference/TAEJANG_PUBLIC_WEB_BRIEF.md)를 먼저 읽고 공개 수준·표현 기준을 준수합니다.
- 게시 전 사실관계, 날짜, 연락처, 인증·선정 표기, 로고 사용 승인 여부를 확인합니다.
- 태장 공식 정보와 대외 표현 기준은 별도 기준 문서로 관리하며, 홈페이지 문구는 그 문서를 우선합니다.
- 법률·고용·인증 관련 문구는 단정 표현을 피하고 필요한 경우 담당자 또는 공식 자료로 재확인합니다.
- 홍보는 단순 노출이 아니라 태장의 고용, 관리, 품질, 제품, 서비스와 성과를 증명하는 활동이어야 합니다.

## 업무 플랫폼 개발 원칙

- 첫 운영 모듈은 홍보 모듈입니다.
- 직원, 팀장, 운영총괄, 대표이사의 역할에 따라 대시보드와 접근 범위를 구분합니다.
- 관리자가 보완을 요청해 직원에게 되돌아온 상태는 직원 화면에서 `보완 필요`로 표시합니다.
- 홍보 모듈이 실제 운영에서 안정화된 뒤 공통 승인·알림 기반과 물류·재고·발주 등 다음 모듈로 확장합니다.
- 외주는 최대한 지양하고 내부 직원, 김형철 운영총괄, ChatGPT와 Codex를 중심으로 개발·검증합니다.
- 회계·급여·금융·법정 신고 등 고위험 영역은 검증된 전문 서비스와 연동하거나 필요한 범위만 외부 검토를 받습니다.
